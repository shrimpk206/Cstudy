# 03. AI에게 에러를 다시 물어보는 법

## 핵심 한 줄

vibe coding의 80%는 "AI가 만든 코드 → 에러 발생 → AI에게 다시 던지기"의 반복입니다. **잘 던지는 사람**이 결과물 품질을 결정합니다.

---

## 나쁜 프롬프트 vs 좋은 프롬프트

### ❌ 나쁨
> "안 돼"
> "에러 났어"
> "이상해"
> "고쳐줘"

AI는 본인 머릿속의 에러를 못 봅니다. 추측만 하게 됩니다.

### ✅ 좋음 — 5가지를 같이 던지세요

1. **에러 메시지 전체** (스택 트레이스 포함)
2. **방금 어떤 동작을 했는지** (버튼 클릭? 페이지 로드? 명령어?)
3. **에러가 난 파일/줄의 코드** (관련 부분)
4. **기대했던 동작**
5. **이미 시도해본 것** (있다면)

---

## 표준 템플릿

```
방금 [어떤 동작]을 했더니 다음 에러가 났어:

[에러 메시지 통째로 붙여넣기]

관련 코드 ([파일명]:[줄]):
```[언어]
[그 부분 코드]
```

내가 기대한 건: [기대 동작]
실제로 일어난 건: [실제 동작]

[있다면] 이미 [무엇을] 시도했는데 그것도 안 됐어.
```

---

## 실전 예시

### 예시 1 — TypeError

❌ 나쁨:
> name이 undefined라고 떠. 고쳐줘.

✅ 좋음:
> 사용자 목록 페이지 로드하니까 다음 에러:
> 
> ```
> TypeError: Cannot read properties of undefined (reading 'name')
>     at UserCard (UserCard.tsx:8:24)
> ```
> 
> UserCard.tsx:8 코드:
> ```jsx
> return <h3>{user.name}</h3>
> ```
> 
> API에서 가끔 user가 null로 오는 것 같아. user가 없을 때 안전하게 처리해줘. UserCard에서 처리할지, 부모에서 필터링할지는 너가 판단.

→ AI가 정확히 짚어 고침. "어떻게 처리할지는 판단" 같이 던지면 더 좋은 답.

### 예시 2 — 의도와 다른 동작 (에러는 안 남)

> 폼 제출 버튼을 누르면 페이지가 새로고침되면서 입력값이 사라져. 새로고침 없이 콘솔에 입력값만 찍히게 해줘. 현재 코드 ([App.jsx:15-25]):
> 
> ```jsx
> [코드 붙여넣기]
> ```

→ 에러가 없어도 "기대 vs 실제"가 명확하면 AI가 잘 고침. 흔한 답: `e.preventDefault()` 누락.

### 예시 3 — 같은 에러가 반복됨

> 아까 useState 추가하라고 해서 추가했는데도 같은 에러 또 남:
> 
> ```
> [에러]
> ```
> 
> 현재 코드:
> ```
> [코드]
> ```
> 
> 무슨 가정이 틀린 것 같아?

"무슨 가정이 틀렸나?"는 마법의 한 줄입니다. AI가 첫 시도의 가설을 의심하게 만듭니다.

---

## 에러 메시지를 그대로 붙이는 게 왜 중요한가

AI가 사람이 요약한 메시지보다 **원문 에러**에서 훨씬 더 많은 단서를 찾습니다.
- 줄 번호, 파일 경로
- 정확한 에러 클래스 이름
- 라이브러리/프레임워크 이름

원문에 적힌 단서를 사람이 가공하면 정보가 사라집니다.

---

## "이미 시도한 것" 알리기 — 시간 절약

AI는 종종 같은 답을 반복합니다. "이건 해봤다"를 알려주면 다른 가설로 갑니다:

> 이 에러:
> ```
> [에러]
> ```
> 
> 이미 시도해본 것:
> - npm install 다시 함 → 그대로
> - node_modules 삭제 후 재설치 → 그대로
> - import 경로 확인 → 맞음
> 
> 다른 원인 가설 줄 수 있어?

---

## 화면 / 로그를 그대로 보여주기

스크린샷, 콘솔 로그 전체, 네트워크 탭 응답 등을 그대로 첨부하면 AI 진단력이 훨씬 좋아집니다. (Claude/Cursor는 이미지 첨부 가능)
