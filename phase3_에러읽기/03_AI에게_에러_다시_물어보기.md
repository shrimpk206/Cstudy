# 03. AI에게 에러 다시 묻기 🩺

## 한 줄로 말하면

**의사한테 진료받는 것**과 똑같아요.
"아파요"만 말하면 의사도 모르죠. **자세히 말해야** 약을 처방해줘요.

---

## 🩺 의사 비유

병원 가서:

❌ "아파요"
의사: "어디가요? 언제부터요? 어떻게 아파요?"

✅ "어제부터 머리 오른쪽이 욱신거려요. 잠도 잘 못 잤고요. 약 먹었는데 안 들어요."
의사: "아 편두통 같네요. 이 약 드셔보세요."

vibe coding에서 AI한테 에러 던지는 것도 똑같아요!

---

## 🚫 나쁜 프롬프트 vs ✅ 좋은 프롬프트

### ❌ 의사 곤란 시리즈

> "안 돼"
> "에러 났어"
> "이상해"
> "고쳐줘"

AI는 본인 화면 못 봐요. 추측만 하게 돼요.

### ✅ 좋은 프롬프트의 5가지 재료 🍱

도시락 싸듯이 5가지 챙겨서 던지세요:

1. 🚨 **에러 메시지 전체** (스택 트레이스 포함)
2. 🎬 **방금 어떤 동작을 했는지** (버튼 클릭? 페이지 로드?)
3. 📝 **에러가 난 파일/줄의 코드**
4. 🎯 **기대했던 동작**
5. 🔄 **이미 시도해본 것** (있으면)

---

## 📋 표준 템플릿 — 복붙해서 쓰세요

```
방금 [어떤 동작]을 했더니 에러:

[에러 메시지 통째로 붙여넣기]

관련 코드 ([파일명]:[줄]):
```[언어]
[그 부분 코드]
```

기대: [기대 동작]
실제: [실제로 일어난 일]

[있다면] 이미 [무엇을] 시도했는데 그것도 안 됐어.
```

---

## 🎬 실전 예시 1: TypeError

### ❌ 나쁨

> name이 undefined라고 떠. 고쳐줘.

### ✅ 좋음

> 사용자 목록 페이지 로드 시 다음 에러:
> 
> ```
> TypeError: Cannot read properties of undefined (reading 'name')
>     at UserCard (UserCard.tsx:8:24)
> ```
> 
> UserCard.tsx:8 코드:
> ```jsx
> return <h3>{user.name}</h3>
> ```
> 
> API에서 가끔 user가 null로 오는 듯. user 없을 때 안전 처리해줘. UserCard에서 처리할지, 부모에서 필터링할지는 너가 판단.

→ AI가 정확히 짚어 고쳐요!

---

## 🎬 실전 예시 2: 의도와 다른 동작

> 폼 제출 버튼 누르면 페이지 새로고침되면서 입력값 사라짐. 새로고침 없이 콘솔에 입력값만 찍히게 해줘. 현재 코드 ([App.jsx:15-25]):
> 
> ```jsx
> [코드 붙여넣기]
> ```

→ 에러가 없어도 **"기대 vs 실제"가 명확**하면 AI가 잘 고쳐요. 흔한 답: `e.preventDefault()` 누락.

---

## 🎬 실전 예시 3: 같은 에러 반복

> 아까 useState 추가하라고 해서 추가했는데도 같은 에러:
> 
> ```
> [에러]
> ```
> 
> 현재 코드:
> ```
> [코드]
> ```
> 
> **무슨 가정이 틀린 것 같아?**

✨ **"무슨 가정이 틀렸나?"는 마법의 한 마디!** AI가 첫 가설을 의심하게 만들어요.

---

## 🤔 왜 에러 메시지를 그대로 붙여야 할까?

AI는 사람이 요약한 메시지보다 **원본 에러**에서 훨씬 더 많은 단서를 찾아요:

- 🔢 줄 번호, 파일 경로
- 🏷️ 정확한 에러 클래스 이름
- 📚 라이브러리/프레임워크 이름

원본의 단서를 **사람이 가공하면 정보가 사라져요.**

---

## 🔄 "이미 시도한 것" 알리기 — 시간 절약

AI는 종종 같은 답을 반복해요. "이건 해봤다"를 알려주면 다른 가설로 가요:

> 이 에러:
> ```
> [에러]
> ```
> 
> 이미 시도한 것:
> - npm install 다시 함 → 그대로
> - node_modules 삭제 후 재설치 → 그대로
> - import 경로 확인 → 맞음
> 
> 다른 원인 가설 줄 수 있어?

## 📸 화면이나 로그 그대로 보여주기
