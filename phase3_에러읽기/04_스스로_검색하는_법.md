# 04. 스스로 검색하는 법

## 핵심 한 줄

AI가 5분 동안 같은 답을 반복하면 본인이 검색하는 게 빠릅니다. **검색 잘 하는 사람 = 더 나은 vibe coder.**

---

## 1) 검색어 만드는 공식

```
[에러 메시지 핵심 부분] [라이브러리/프레임워크 이름]
```

예시:
- ❌ "리액트 안 돼" → 99만 결과 중 본인 케이스 0
- ✅ `Cannot read properties of undefined react useState` → 정확한 케이스
- ✅ `nextjs 14 environment variable not loading` → Next.js 14 한정 결과

**팁:**
- 본인 변수명/파일명은 제외 (다른 사람한텐 의미없음)
- 버전 정보 포함 (`React 18`, `Next.js 14`, `Python 3.11`)
- 영어로 검색하면 결과가 압도적으로 많음

## 2) 어디서 검색하나

| 사이트 | 무슨 정보 | 언제 |
|---|---|---|
| **Google** | 일반 | 가장 먼저 |
| **Stack Overflow** | Q&A | 비슷한 사람이 이미 물어봤을 확률 70% |
| **GitHub Issues** | 라이브러리 버그 | 라이브러리 자체에 버그가 있을 때 |
| **공식 문서** | 정확한 사용법 | API 사용법 헷갈릴 때 |
| **MDN (Mozilla)** | JavaScript/HTML/CSS 표준 | 웹 표준 기능 (Array.map 등) |

## 3) Stack Overflow 답변 읽는 법

- **녹색 체크 표시**: 질문자가 정답으로 채택한 답변
- **점수 (좌측 숫자)**: 사람들이 좋다고 추천한 수 — 정답보다 점수 높은 답이 있으면 그게 더 정확할 때 많음
- **댓글**: 답변 아래 작은 글씨 — "이거 옛날 방식임", "이건 위험함" 같은 보충
- **날짜**: 5년 전 답변은 옛날 문법일 수 있음. 최신 답변 우선

## 4) 공식 문서 읽기

처음엔 어렵게 느껴지지만 vibe coding에 가장 신뢰할 수 있는 정보:

- **React**: react.dev
- **Next.js**: nextjs.org/docs
- **Node.js**: nodejs.org/docs
- **Python**: docs.python.org
- **MDN**: developer.mozilla.org (Array, Object, fetch 등 JavaScript 표준)

**팁:** 라이브러리 README에 "Quick Start" 또는 "Getting Started" 섹션이 거의 다 있습니다. 거기부터.

## 5) GitHub Issues 활용

라이브러리 자체에 버그가 의심되면:
1. 라이브러리 GitHub 페이지 → Issues 탭
2. 검색창에 에러 메시지 일부 붙여넣기
3. **Closed** 이슈도 같이 보세요 — 이미 해결된 케이스에 답이 있음

## 6) AI를 검색 도구로 쓰기

ChatGPT, Claude, Perplexity 등에 직접 검색을 시킬 수 있습니다:

> "Next.js 14에서 환경변수를 추가했는데 클라이언트에서 undefined로 나와. 이게 보안 정책 때문인지, 설정 누락인지, 알려진 버그인지 알려줘. Next.js 공식 문서 인용해서."

이때 **답을 그대로 믿지 말고 한 번 검증**하세요. AI는 종종 옛날 정보를 줍니다.

## 7) 본인 코드에서 직접 디버깅

검색 전에 한 번 시도할 가치가 있는 것:

### console.log 폭격
```javascript
console.log("여기까지 옴")
console.log("user:", user)
console.log("typeof user:", typeof user)
console.log("user keys:", Object.keys(user || {}))
```
값이 본인 예상과 다른 곳을 찾는 가장 빠른 방법.

### 이진 탐색
코드 절반을 주석 처리해서 어느 쪽에 문제가 있는지 좁히기. 그 절반을 또 절반으로...

### 최소 재현 예제
문제 부분만 따로 작은 파일에 떼어내서 재현. 떼어내는 과정에서 원인이 드러나기도 합니다.

---

## 검색 → 적용 흐름

1. 에러 메시지에서 핵심 부분 추출
2. 영어로 검색 (라이브러리 이름 포함)
3. Stack Overflow 또는 GitHub Issue에서 비슷한 케이스 발견
4. 거기 솔루션을 **그대로 복붙하지 말고**, 본인 코드에 어떻게 적용할지 AI에게 물어보기:
   > "이 Stack Overflow 답[링크/내용]이 우리 케이스랑 맞는 거 같은데, 우리 코드에 어떻게 적용해? 코드: [본인 코드]"

이게 검색과 vibe coding을 연결하는 황금 패턴입니다.
