# 04. 스스로 검색하는 법 🔎

## 한 줄로 말하면

**보물찾기**예요.
AI가 5분째 같은 답 주면? 본인이 직접 단서 찾는 게 빨라요.

검색 잘 하는 사람 = **더 나은 vibe coder.**

---

## 🔎 검색어 만드는 공식

```
[에러 메시지 핵심 부분] [라이브러리/프레임워크 이름]
```

### 비교

| 검색어 | 결과 |
|---|---|
| ❌ "리액트 안 돼" | 99만 개 결과 중 본인 케이스 0 |
| ✅ `Cannot read properties of undefined react useState` | 정확한 케이스 |
| ✅ `nextjs 14 environment variable not loading` | Next.js 14 한정 |

### 💡 팁

- **본인 변수명/파일명은 제외** (다른 사람한텐 무의미)
- **버전 정보 포함** (`React 18`, `Next.js 14`, `Python 3.11`)
- **영어로 검색** (결과가 압도적으로 많아요!)

---

## 🗺️ 어디서 검색하나요?

| 사이트 | 무슨 정보 | 언제 |
|---|---|---|
| 🔍 **Google** | 일반 | 가장 먼저 |
| 💬 **Stack Overflow** | Q&A | 비슷한 질문이 이미 있을 확률 70% |
| 🐛 **GitHub Issues** | 라이브러리 버그 | 라이브러리 자체 버그 의심 |
| 📚 **공식 문서** | 정확한 사용법 | API 사용법 헷갈릴 때 |
| 📖 **MDN (Mozilla)** | JS/HTML/CSS 표준 | Array.map 같은 표준 기능 |

## 💬 Stack Overflow 답변 평가하기

답변이 여러 개일 때 어떤 걸 믿을까요?

| 신호 | 의미 |
|---|---|
| ✅ **녹색 체크** | 질문자가 정답으로 채택 |
| 🔢 **점수 (좌측 숫자)** | 좋아요 수 |
| 💬 **댓글** | "이거 옛날 방식임" 같은 보충 |
| 📅 **날짜** | 5년 전 답변은 옛날 문법일 수도! |

> 💡 **점수 > 채택**일 때도 있어요! 점수 더 높은 답이 정확할 가능성. 그리고 **최신 답변** 우선.

## 📚 공식 문서 — 가장 신뢰할 수 있는 곳

처음엔 어렵게 느껴지지만 가장 정확해요:

| 분야 | 사이트 |
|---|---|
| React | react.dev |
| Next.js | nextjs.org/docs |
| Node.js | nodejs.org/docs |
| Python | docs.python.org |
| JavaScript 표준 | developer.mozilla.org (MDN) |

> 💡 라이브러리 README에 거의 다 "Quick Start" / "Getting Started" 섹션이 있어요. 거기부터!

---

## 🐛 GitHub Issues — 라이브러리 버그 의심 시

라이브러리 자체에 버그가 의심되면:

1. 그 라이브러리 GitHub 페이지 → **Issues** 탭
2. 검색창에 에러 메시지 일부 붙여넣기
3. **Closed** 이슈도 같이 봐요 — 이미 해결된 케이스에 답이 있음

---

## 🤖 AI를 검색 도구로 쓰기

ChatGPT, Claude, Perplexity에 직접 검색 시킬 수 있어요:

> "Next.js 14에서 환경변수 추가했는데 클라이언트에서 undefined로 나옴. 보안 정책 때문인지, 설정 누락인지, 알려진 버그인지 알려줘. Next.js 공식 문서 인용해서."

> ⚠️ **답을 그대로 믿지 말고** 한 번 검증하세요. AI는 종종 옛날 정보를 줘요.

## 🔧 본인 코드 직접 디버깅 — 검색 전 시도

### console.log 폭격

```javascript
console.log("여기까지 옴")
console.log("user:", user)
console.log("typeof user:", typeof user)
console.log("user keys:", Object.keys(user || {}))
```

값이 본인 예상과 다른 곳 찾는 가장 빠른 방법!

### 이진 탐색

코드 절반을 주석 처리해서 어느 쪽에 문제가 있는지 좁히기. 그 절반을 또 절반으로...

### 최소 재현 예제

문제 부분만 따로 작은 파일에 떼어내서 재현. 떼어내는 과정에서 원인이 드러나기도 해요.

## 🌟 검색 → 적용 황금 흐름

1. 에러 메시지에서 **핵심 부분 추출**
2. **영어로** 검색 (라이브러리 이름 포함)
3. Stack Overflow / GitHub Issue에서 비슷한 케이스 발견
4. 거기 솔루션을 **그대로 복붙하지 말고** AI에게 물어봐요:
   > "이 Stack Overflow 답[링크]이 우리 케이스랑 맞는 거 같은데, 우리 코드에 어떻게 적용해? 코드: [본인 코드]"

이게 검색과 vibe coding을 연결하는 **황금 패턴**이에요!
