# 01. 좋은 명세 쓰는 법

## 핵심 한 줄

vibe coding 결과물 품질 = **첫 명세의 정확도**. 시작이 모호하면 끝까지 표류합니다.

---

## 나쁜 명세 vs 좋은 명세

### ❌ 나쁨
> "할 일 관리 앱 만들어줘"

AI가 받는 정보:
- 무슨 기능들이 들어가는지: ?
- 어떻게 생겼는지: ?
- 데이터 어디에 저장: ?
- 사용자가 누구: ?
- 성공이 뭔지: ?

→ AI가 추측으로 채움. 본인 의도와 빗나가는 결과.

### ✅ 좋음
> 다음 사양의 할 일 관리 웹앱을 만들어줘.
> 
> **사용자:** 개인이 혼자 쓰는 용도. 로그인 없음.
> 
> **기능:**
> 1. 할 일 추가 (텍스트만, 우선순위 같은 부가정보 X)
> 2. 할 일 완료 표시 (체크박스)
> 3. 완료된 항목 일괄 삭제 버튼
> 4. 새로고침해도 데이터 유지
> 
> **저장:** 브라우저 localStorage. 백엔드 없음.
> 
> **기술:** React + TypeScript + Tailwind CSS.
> 
> **UI:** 단일 페이지. 상단 입력창, 아래 목록. 다크 모드는 없어도 됨.
> 
> **성공 기준:**
> - 텍스트 입력 후 Enter로 추가 가능
> - 체크박스로 완료 표시 시 즉시 회색+취소선
> - 새로고침 후에도 모든 데이터 그대로
> 
> 시작은 단일 컴포넌트로, 100줄 이내로 부탁.

---

## 좋은 명세의 7가지 요소

### 1. 사용자 — 누가 쓰나
- 본인만 쓸 건지 / 여러 명이 쓸 건지
- 로그인 필요 여부
- 모바일 / 데스크톱 어느 쪽 우선

### 2. 핵심 기능 — 번호 매겨서 나열
- 5개 이하로 압축 (많으면 단계별로 나누기)
- 각각 한 줄로

### 3. **명시적으로 빼는 것** ← 매우 중요
- "검색 기능은 지금은 빼줘"
- "다국어 지원 안 함"
- "관리자 페이지 없음"

AI는 시키지 않으면 자꾸 부풀립니다. **하지 말 것**을 알려주는 게 더 효과적일 때가 많습니다.

### 4. 데이터 — 어디에 저장
- localStorage / IndexedDB (브라우저 안)
- 파일 (Node.js 백엔드)
- 데이터베이스 (Postgres, MongoDB, SQLite, Firestore...)
- 지금은 메모리만 (재시작하면 날아감)

### 5. 기술 스택
- 언어: JavaScript / TypeScript / Python
- 프레임워크: React / Next.js / Vue / Express / FastAPI
- 스타일: Tailwind / CSS / 안 정함

**모르겠다면**: "추천해줘. 내가 vibe coding으로 진행할 건데, AI가 잘 만들어내는 흔한 스택으로."

### 6. UI — 어떻게 생겼는지
- 화면 개수
- 페이지 구조 (상단 헤더, 좌측 사이드바, 본문...)
- **참고 이미지 / 사이트** 있으면 첨부 ("Notion처럼", "Linear처럼")
- 색감, 다크 모드 여부

### 7. 성공 기준 — 뭐가 되면 끝인가
체크 가능한 형태로:
- "Enter 키로 추가됨"
- "새로고침해도 데이터 남음"
- "에러 없이 빌드됨"

이게 없으면 "다 됐어요"라는 AI 말을 검증할 방법이 없습니다.

---

## 명세를 한 번에 다 적지 마세요

긴 명세를 한 번에 던지면 AI가 핵심을 놓칩니다. 추천 흐름:

1. **첫 메시지**: 기능 1~2개 + 핵심 사용자 + 스택만
2. AI가 만든 결과 확인
3. 다음 기능 추가 요청
4. 반복

작게 시작 → 동작 확인 → 추가. 이게 vibe coding의 안정적 진행 방식.

---

## 명세를 AI에게 다듬게 하기

자신 없으면 명세 자체를 AI와 같이 만들 수 있습니다:

> 다음을 만들고 싶어: [한 문단 설명]. 
> 본격 작업 전에 이 명세에서 빠진 부분이 뭔지, 모호한 부분이 뭔지 질문해줘. 
> 결정해야 할 것 5개를 추려서 보여줘.

AI가 5개 질문을 던지면 → 답변 → 그 답변까지 합쳐서 다시 명세 정리 요청.

---

## 자가점검

다음 명세에 빠진 부분 3개를 찾아보세요.

> "포트폴리오 사이트 만들어줘. 내 프로젝트들 보여주는 페이지 있고, 연락처도 있고. 멋있게."

<details>

빠진 것 (많음):
- 기술 스택 (Next.js? Astro? 정적 HTML?)
- 프로젝트 데이터를 어떻게 입력/관리하나 (마크다운? JSON 파일? CMS?)
- "멋있게"의 구체적 정의 (참고 사이트, 다크/라이트, 색감)
- 페이지 개수 (하나? 프로젝트별 상세 페이지?)
- 연락처 폼은 실제 메일 발송? 아니면 mailto 링크?
- 모바일 우선 여부
- 배포 (Vercel? GitHub Pages?)
- 다국어 지원 여부

좋은 시작점은: 위 모든 걸 사전에 정해서 명시. 또는 AI에게 "이 명세에서 결정 필요한 부분 5개를 질문해줘"라고 던지기.
